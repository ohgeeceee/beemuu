# BeeEmUu forum

Community threads live as plain markdown files in this directory.
Each thread is one file. You file a thread by opening a pull
request — same workflow as `community/profiles/*.toml`,
`community/dtc_texts.toml`, and `community/hunt/leaderboard.json`.

> **Why PRs, not a web form?** The project is offline-first and
> ships via git. Every other community contribution — DTC texts,
> engine profiles, freeze-frame schemas, oracle patterns, hunt
> scores — works this way. The forum is the same pattern applied to
> prose. No new infrastructure, no auth wall, no DB. GitHub
> credentials are the credentials.

## How to file a thread

1. **Fork** the repo.
2. **Copy** `community/forum/threads/TEMPLATE.md` to
   `community/forum/threads/<your-slug>.md`. Slug should be
   short, lowercase, hyphenated (e.g. `b48-stage1-knock.md`,
   `n52-oil-temp-hunt.md`).
3. **Fill in the frontmatter** (between the `---` markers) and the
   body (markdown).
4. **Run** `python scripts/forum_index.py` from the repo root to
   regenerate `community/forum/index.json`. CI will run this and
   fail the PR if `index.json` is out of date.
5. **Open a PR** titled `forum: <your-slug>`.

## How to reply to a thread

Same as filing — open a PR that edits the existing thread file.
Add your reply under a `## Replies` heading in the thread, with a
`> **<your-github-handle> at <date>:**` prefix. Replies are
appended in chronological order; the maintainer merges them.

There is no fancy threading UI. There is no "1 day ago" timestamp.
The git log is the timeline. If that bothers you, this is not the
right forum.

## Thread template

See [`threads/TEMPLATE.md`](threads/TEMPLATE.md). Required
frontmatter:

```yaml
---
title: "..."           # Human-readable title, <=120 chars
author: <github-user>   # Your GitHub handle
date: YYYY-MM-DD        # Today's date
tags: [<comma-list>]    # e.g. [b58, stage-1, knock, boost]
related_dtcs: [<hex>]   # Optional: fault codes discussed (hex, no 0x)
config:                 # Optional: shared config in this thread
  path: community/profiles/...
  checksum: sha256:...
---
```

## What "share a config" means here

Config fields are flat (`config_path`, `config_checksum`) so the
TOML parser can read them without nested-table gymnastics. Use empty
strings if there's no config to share:

```yaml
config_path: community/profiles/b48_stage1.toml
config_checksum: "sha256:abc123..."
```

Or:

```yaml
config_path: ""
config_checksum: ""
```

The thread is for prose, screenshots, logs, and discussion. The
data is in git. This split is the whole point of the offline-first
model.

## Anti-features (deliberate)

- **No upvotes / karma / badges.** The Parameter Hunt game covers
  the gamification use case; forum posts are for help and sharing,
  not status.
- **No private messages.** If it's worth saying, say it on a
  thread.
- **No email notifications.** Watch the repo.
- **No bot / AI replies.** Replies are from humans with cars.
- **No attachments larger than a markdown image.** Big logs go in
  git (or a gist linked from the thread), not in the forum.

## Index

`index.json` is auto-generated by `scripts/forum_index.py`. Don't
edit it by hand — the script overwrites it on every run. CI runs
the script on every PR and fails the build if the committed
`index.json` doesn't match what the script would produce. So just
edit the thread file, run the script, commit both, push.

## When this model breaks

The plan in `docs/FORUM_PLAN.md` is explicit: stay on Option A
(static markdown) until **any one** of these happens:

- More than ~200 threads total.
- A single thread with more than ~20 replies.
- Search-quality complaints in 3+ separate threads.

When one of those triggers, escalate to Option B (FastAPI +
SQLite + GitHub OAuth) and write a one-time importer that walks
this folder and seeds the new DB. The data shape is the same
(title, author, date, body, tags) so the cutover is mechanical.
