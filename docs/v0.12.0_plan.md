# v0.12.0 Plan — "Fault Memory"

*Drafted 2026-07-21 against `main` @ fd381a9 (v0.11.0 "Share the Trace" closed; #131/#135/#136/#138/#142 all merged).*

## Premise

Closing the DTC panel is closing the diagnosis. An owner reads faults, sees "2A82 — VANOS intake solenoid fault", clears them, drives away — and the next time the same code appears, the app has no idea. The data is right there in the read pipeline (DTC code, module address, status, status_text, freeze frame, odometer if available); nobody writes it down.

v0.12.0 closes that gap by persisting every DTC read to a local JSONL log and surfacing a **Fault Memory** panel: "this DTC has appeared N times over the past K days on this car". Local-only, no cloud, no privacy surprise. The persistence file lives under the same `~/beeemuu-exports/` directory the v0.6.0 export pipeline already uses.

## Honest ceiling

- **Local-only.** No upload, no sync, no telemetry. The user owns the file and can delete it.
- **One car per file.** Identified by VIN where available; "no VIN" entries are flagged in the UI rather than silently merged.
- **Append-only JSONL.** Each DTC read is a single line. No migration story needed — the schema is forward-compatible (new fields can be added without breaking old files).
- **Opt-in recording.** A toggle in Settings / a new panel header turns recording on. Off by default so existing users don't suddenly find a new file in `~/beeemuu-exports/` without explanation.
- **Read-only.** History is queried, never written back to the ECU. No commands touch the bus.

## What we will NOT do (this cycle)

- **Cloud sync, multi-device merge, account-bound history.** Privacy + infrastructure ceiling. Revisit if a user need materialises.
- **Predictive "you'll likely see 2A82 again next month" ML.** Out of scope. The data lands first; the modelling is a future cycle.
- **Edit / annotate / delete individual history entries.** The user can delete the whole file. Single-entry editing is a slippery slope (audit trail questions) — defer.
- **Export history as CSV.** A nice-to-have but the per-entry shape is dense; defer until users ask.
- **Pre-emptive recording from background scans.** Only the explicit `read_faults` invocation records, so the history reflects what the user actually looked at — not silent background chatter.

`DtcHistorySummary` is a new serializable struct:
```rust
pub struct DtcHistorySummary {
    pub entries: Vec<DtcHistoryEntry>,  // sorted by last_seen descending
    pub total_lines: usize,
    pub file_path: String,              // for the UI to surface ("stored at ~/...")
}
pub struct DtcHistoryEntry {
    pub code: String,
    pub address: u8,
    pub status_text: String,
    pub text: String,
    pub first_seen_iso: String,
    pub last_seen_iso: String,
    pub occurrences: usize,
}
```

The 60 s dedup window is the "concurrent read of the same module" guard — not a permanent id. Each new read after 60 s counts as a new occurrence.

### Slice 3 — `dtc_history.js` pure module + tests (Tier A, frontend only)

Pure data → JS module that:
- Wraps the three Tauri commands (`recordDtcRead`, `queryDtcHistory`, `clearDtcHistory`).
- Exposes a higher-level API: `recordCurrentRead(vin, address, lastDtcs)` — called by `readFaults` after a successful read.
- Dual export (CommonJS + `window.beeemuuDtcHistory`) so it tests under `node --test`.
- 100% unit-tested with a tiny in-memory JSONL store (no fs in tests).

### Slice 4 — recording wired into `readFaults` + recording toggle in Settings (Tier A, frontend)

- `readFaults()` in `main.js` calls `dtc_history.recordCurrentRead(vin, selectedAddress, lastDtcs)` after the read resolves (success or empty — empty still records "scanned, no faults" so the timeline shows the user looked).
- New "Fault Memory" panel under the DTC tab (or as a third tab if the tab bar has room — propose, no churn).
- Toggle in Settings: "Record DTC reads to local history". Off by default; first-time-on shows a one-line explainer.
- VIN is read from the existing `info-vin` text; if absent, history entries are tagged `"vin": null` and the UI warns "history is grouped per-car only when VIN is known".

### Slice 5 — "recurring DTC" callout in the DTC panel (Tier A, frontend)

When `lastDtcs.length > 0`, query `queryDtcHistory` for the current VIN and surface a small banner under the DTC table:

> ⚠ 2A82 has appeared 3 times in the past 14 days (last: today, 09:42)

This is the **single highest-value UI moment** of the cycle: it's the answer to the question "is this the same fault I cleared last week?". Pure read, frontend only.

## Why "Fault Memory" as the cycle name

Cycle names track the user promise. v0.11.0 made BeeEmUu something you can hand to someone else. v0.12.0 makes it something that **remembers your car** between sessions. The name is deliberately the mental model an owner has of the app after a few months: "I open BeeEmUu and it already knows what faults I've been chasing".
