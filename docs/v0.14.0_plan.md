# v0.14.0 Plan — "Live CAN"

*Drafted 2026-07-23 against `main` @ c701157 (v0.13.0 "Real Reads, Real
Long" closed; #143–#155 merged; slice 3 of v0.13.0 promoted to its own
cycle per the corrected v0.13.0 plan note: "Add a SocketCAN or OBDLink
STN transport. That's a v0.14.0+ follow-up").*

## Premise

E-series owners can't read their live data today — KWP2000 local IDs
are unmapped in open sources
(`docs/ROADMAP_ISSUES.md` "Alternative to KWP2000 local IDs";
`research/bmw_diag_dim07_local_ids.md`). But BMW ECUs **broadcast** the
data anyway on the raw CAN bus at 500 kbit/s. The bytes are free, we
just don't listen.

v0.14.0 adds a `Live CAN` transport mode that:
- Switches an OBDLink SX USB cable into raw CAN mode at 500 kbit/s
- Filters by CAN ID to reduce noise (only 0x0AA, 0x1D0, 0x545, 0x0CE,
  0x130, 0x316 — the 6 known broadcast IDs from
  `docs/ROADMAP_ISSUES.md` "Known Broadcast Frames")
- Decodes each frame into a typed value (RPM, coolant °C, oil temp °C,
  wheel speeds, vehicle speed, battery voltage)
- Surfaces them as a new "Live Gauges" panel using the existing
  `src/js/gauges.js` widget

**Hardware choice (v0.14.0 scope):** OBDLink SX on an E46 (default per
the planning conversation; if the user's bench changes, the decoder
+ UI + simulator path is unchanged and only slice 5's cable-specific
Tauri-command wiring changes).

## Honest ceiling (what this cycle is NOT)

- **Not a K+DCAN cable firmware change.** The existing
  `src-tauri/src/transport/kdcan.rs` keeps doing KWP2000-over-CAN for
  diagnostic sessions. The new `transport/can_listener.rs` is a
  **separate** transport that runs in parallel with KWP, not instead
  of it. Real-world workflows need both: KWP for fault reads, raw CAN
  for live data, simultaneously.
- **Not a new crate dependency for the OBDLink SX path.** The SX
  presents as a USB-CDC ACM serial device; the existing
  `serialport = "4"` crate in `Cargo.toml` covers it. STN1110
  protocol commands go over the standard serial handle. No
  `nb-sys` / `peak-can` / `socketcan` dep is added in v0.14.0.
- **Not a chassis-agnostic rollout.** The 6 known broadcast IDs are
  confirmed on E46 per `docs/ROADMAP_ISSUES.md`. E9x and E6x frame
  layouts are documented as **needing verification** (especially
  0x545 oil temp — "E46 confirmed; E9x needs verification"). v0.14.0
  tests on E46; the harness doc (slice 7) tells E9x/E6x users what
  to report back.
- **Not a port-listener (TCP/UDP).** Pure serial USB. The v0.15.0
  follow-up is a SocketCAN path for users who already have a PEAK
  PCAN-USB; the v0.15.1 follow-up after that is a Windows SocketCAN
  equivalent.

## What we will NOT do (this cycle)

- **Touch the K+DCAN transport** (`src-tauri/src/transport/kdcan.rs`).
  The new `can_listener.rs` is additive. The two transports run side
  by side; KWP diagnostic sessions keep using `kdcan.rs`.
- **Touch `src-tauri/src/protocol/`.** UDS/KWP request builders stay
  unchanged. The CAN listener is a raw-frame consumer with no UDS
  or KWP semantics.
- **Re-architect the gauges system.** `src/js/gauges.js` is a
  reusable widget (already used by the existing Live Data tab). The
  new "Live Gauges" panel reuses it without modification.
- **Add a new crate.** `serialport` already covers the OBDLink SX.
- **Run the listener by default.** Starting a background thread that
  holds open a USB serial port is a side effect the user should
  consent to. The panel is off by default; user clicks "Start CAN
  listener" in the panel header to enable it.

### Slice 2 — `src/js/can_decoders.js` pure decoder module + tests (Tier A, frontend)

Pure functions. Each takes an 8-byte CAN frame, returns a typed value:

| CAN ID | Decoder | Output | Scale |
|---|---|---|---|
| 0x0AA | `decodeRpm(frame)` | `number` (RPM) | big-endian u16 at bytes 0-1, scale 0.25 |
| 0x0AA | `decodeThrottle(frame)` | `number` (% 0-100) | byte 6, scale 0.3922 |
| 0x1D0 | `decodeCoolant(frame)` | `number` (°C) | byte 0, signed offset -48 |
| 0x1D0 | `decodeAmbientTemp(frame)` | `number` (°C) | byte 1, signed offset -48 |
| 0x545 | `decodeOilTemp(frame)` | `number` (°C) | byte 1, signed offset -48 |
| 0x0CE | `decodeWheelSpeeds(frame)` | `[fl, fr, rl, rr]` (km/h) | 4× big-endian u16, scale 0.0625 |
| 0x130 | `decodeVehicleSpeed(frame)` | `number` (km/h) | byte 0, scale 0.5 |
| 0x316 | `decodeBatteryVoltage(frame)` | `number` (V) | byte 0, scale 0.1 + 6.0 offset |

Scales from `docs/ROADMAP_ISSUES.md` "Real Car / Alternative to
KWP2000 local IDs" + the BMW E46 DME service manual conventions. The
scale constants are extracted to named consts at the top of the file
so a future E9x-discrepancy investigation can change them in one
place.

Tests with known-good fixtures:
- 0x0AA frame `00 00 1F FF 00 00 4A 00` → 2046 RPM, ~28.6% throttle
- 0x1D0 frame `5A 60 FF FF FF FF FF FF` → 42 °C coolant, 48 °C ambient
- 0x545 frame `00 5A FF FF FF FF FF FF` → 42 °C oil
- 0x130 frame `64 00 ...` → 50 km/h
- 0x316 frame `5A 00 ...` → 14.0 V battery

Dual export (CommonJS + `window.beeemuuCanDecoders`), same pattern
as the other v0.12.0 / v0.13.0 pure modules. Pure data → numbers, no
DOM, no Tauri IPC.

### Slice 3 — `src/js/live_gauges.js` panel (Tier A, frontend)

New "Live Gauges" panel that renders 6 gauges using the existing
`src/js/gauges.js` `Gauge` widget:

| Gauge | CAN ID | Min | Max |
|---|---|---|---|
| RPM | 0x0AA | 0 | 8000 |
| Coolant | 0x1D0 | -10 | 130 |
| Oil temp | 0x545 | -10 | 150 |
| Vehicle speed | 0x130 | 0 | 250 |
| Battery voltage | 0x316 | 10 | 16 |
| Throttle | 0x0AA | 0 | 100 |

Layout: 2×3 grid of `<canvas>` elements, each backed by a
`new Gauge(canvas, { label, unit, min, max })`. The panel header has
a "Start CAN listener" button (off by default per the honest
ceiling). When the listener is off, the gauges show a muted "off"
state; when on, they redraw from cached values every 100ms.

The render loop and the value-cache are pure JS in this file — no
Tauri IPC yet. Slice 6 wires the cache to the real command. This
separation is intentional: the panel is testable as a pure module
under `node --test`, and the slice 6 wiring is a thin integration.

### Slice 4 — `src-tauri/src/transport/sim.rs` broadcast personality (Tier A, simulator only)

Extend the existing `SimTransport` (and `SimCanBus` for the
ISO-TP-side test path) to also **broadcast** raw CAN frames at the
documented rates. This is the test source for slices 2-3 and the
test path for slice 5.

Broadcast schedule (timings approximate, derived from
`docs/ROADMAP_ISSUES.md`):

| CAN ID | Period | Notes |
|---|---|---|
| 0x0AA | 10 ms | DME: RPM, torque, throttle |
| 0x1D0 | 100 ms | DME: coolant, ambient |
| 0x545 | 1000 ms | DME: oil temp, oil pressure |
| 0x0CE | 20 ms | DSC: wheel speeds |
| 0x130 | 100 ms | EGS/DME: vehicle speed, gear |
| 0x316 | 1000 ms | DME/IHKR: battery voltage, charging |

The simulator generates plausible BMW DME-style values that vary
over time: RPM follows a sine wave (idle → redline → idle), coolant
slowly warms up from ambient to operating temp, oil temp follows
coolant with a 5-10 °C offset, vehicle speed is operator-controllable
(via a test-side knob the slice 7 harness doc explains), battery
voltage drifts between 13.5 V and 14.5 V with a sine + noise.

Implementation: a `mpsc::Sender<CanFrame>` channel owned by the
simulator. The broadcast thread sends frames into the channel. The
slice 5 listener reads from the channel.

This is Tier A because the simulator is in-process and the broadcast
loop is just a `std::thread::spawn` writing to `mpsc`. The
`src-tauri/src/transport/` rule for Tier B is about the **production
transport** (KD-CAN, ENET). The simulator lives in the same
directory by convention but is not the trust boundary.

### Slice 5 — `src-tauri/src/transport/can_listener.rs` (Tier B, Rust)

The new transport module. Tier B because it's in `transport/`.

Public surface:
- `pub struct CanFrame { pub id: u16, pub data: [u8; 8], pub t_ms: u64 }`
- `pub enum ListenerMode { Simulator, OBDLinkSx { port_name: String } }`
- `pub struct CanListener { mode: ListenerMode, latest: Arc<Mutex<HashMap<u16, CanFrame>>>, thread: Option<JoinHandle> }`
- `impl CanListener { pub fn start(mode: ListenerMode) -> Result<Self>; pub fn stop(self) -> Result<()>; pub fn latest(&self) -> HashMap<u16, CanFrame> }`

The `Simulator` mode spawns a thread that reads from the slice-4
mpsc channel and updates the shared `latest` map. The
`OBDLinkSx` mode opens the serial port with `serialport::new(...)`,
sends the STN1110 init sequence (`STI` + `STP 33` for raw CAN, `SP
6` for 500 kbit/s), and polls `STMA` / `STM` for received frames. The
exact STN1110 command sequence is documented in slice 7's harness
doc.

The frame-update loop: receive 8 bytes, parse the CAN ID from
`bytes[0..4]` (STN extended format), copy `bytes[4..12]` into a
`CanFrame.data`, update `latest[frame.id] = frame`.

`tests/`:
- Unit test for the `Simulator` mode: spawn the listener, push 3
  frames through the slice-4 channel, assert `latest()` returns them
  all.
- Unit test for the OBDLink SX frame parser: a scripted serial
  stream with a known STN `STM` response, assert the parsed
  `CanFrame` has the right ID and data bytes.

### Slice 6 — `src-tauri/src/commands.rs` Tauri command surface (Tier B, Rust)

Three new async commands:
- `pub async fn start_can_listen(state, mode: ListenerMode) -> Result<(), String>`
- `pub async fn stop_can_listen(state) -> Result<(), String>`
- `pub async fn get_latest_can_frames(state) -> Result<HashMap<u16, CanFrame>, String>`

The first two hold an `Arc<CanListener>` in `AppState`; the third
reads from it. The async wrapper around `start_can_listen` uses
`spawn_blocking` for the serial-port open (which can block on
Windows driver load).

`tests/async_commands.rs` allowlist test: the three new commands
are async, so the test passes without any allowlist change. The
allowlist test is a positive list + a "any non-allowlisted command
must be async" rule; new async commands don't need to be in the
allowlist.

### Slice 7 — Frontend integration: Live Gauges panel wiring (Tier A, frontend)

The slice-3 panel's `setInterval(render, 100ms)` call site gets
wired to the new `get_latest_can_frames` Tauri command. The
integration is a thin shim:

```js
async function pollLatest() {
  const frames = await invoke('get_latest_can_frames');
  for (const [id, frame] of Object.entries(frames)) {
    const value = window.beeemuuCanDecoders.decodeFor(id, frame.data);
    gauges[id]?.update(value);
  }
}
```

The `Start CAN listener` button calls
`invoke('start_can_listen', { mode: { Simulator: {} } })` for now;
slice 5's OBDLink SX path adds a UI dropdown for cable selection in
a follow-up.

## Why "Live CAN" as the cycle name

Cycle names track the user promise. v0.11.0 made sessions
shareable. v0.12.0 made the app remember. v0.13.0 made real
reads work (the KWP timeout fix). **v0.14.0 makes live data work**
— the bytes BMW broadcasts for free on the bus, surfaced to the
app at last, without users having to find unmapped KWP2000 local
IDs in forum posts. "Live CAN" names the user-visible outcome
(real-time gauges on the Live Gauges panel), not the mechanism
(raw CAN frame transport).
