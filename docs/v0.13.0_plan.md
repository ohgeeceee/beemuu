# v0.13.0 Plan — "Real Reads, Real Long"

*Drafted 2026-07-22 against `main` @ 7dd7fd1. Revised 2026-07-23 after
the original plan's slice-2 premise turned out to be wrong on re-ground.*

## What changed in this revision

The first draft (committed at `f413f61`, merged via #150) claimed
that ISO-TP multi-frame was implemented but not wired into the
production `connect()` paths, and proposed v0.13.0 slice 2 as the
"wire-up" PR. **That premise was wrong.**

On re-grounding against the code, the existing `isotp.rs` module
doc-comment is explicit and authoritative:

> **K+DCAN cable (`kdcan.rs`): never reaches the PC.** The cable is
> an FTDI bridge whose firmware terminates the CAN side; the PC
> speaks complete BMW-framed messages... no ISO-TP machinery in
> `kdcan.rs`.

> **ENET (`enet.rs`): never reaches the PC.** The ZGW gateway
> terminates CAN-side ISO-TP and delivers one reassembled
> diagnostic payload per HSFZ message.

So the production `kdcan.rs::request` and `enet.rs::request` already
return complete payloads — the gateways (FTDI chip, ZGW) handle
ISO-TP termination upstream of the app. ISO-TP is the
**integration point for future raw-CAN transports** (SocketCAN,
OBDLink STN, future DoIP socket) that don't exist yet. The existing
`sim.rs` exercises it end-to-end through `SimCanBus`, and the
existing `vin_read_survives_multi_frame_segmentation` test verifies
the implementation works.

**There is no production wire-up to do.** This revision of the
plan drops slice 2 (the wire-up) and slice 3 (the VIN regression
test that no longer has a failing path to guard) and rescopes the
cycle to the slices that *are* real wins.

## Premise (revised)

Two genuine, user-visible wins have been on the ROADMAP since v0.3.0
without shipping:

- **KWP2000 slow-module timeout fix** (🟢 Ready, line 52): the
  hardcoded 1000ms deadline in `kdcan.rs::request` times out on
  real E-series CIC/CAS modules. Small, well-scoped, real bug.
- **E-series CAN broadcast frame decoder** (🟡, line 45): the
  bytes are already on the bus from the transport; the renderer
  just doesn't decode them. Pure frontend. For E46 owners, this
  gives them a working tachometer and coolant gauge that the
  current app doesn't show.

Plus the `docs/validation/multi-frame.md` doc, which now earns its
keep by explaining what the `isotp.rs` integration point *is* for
(future raw-CAN transports), how to validate against the simulator's
multi-frame personality today, and why the production code doesn't
need it.

## Honest ceiling

- **No ISO-TP production wiring.** That's correct as-is — the
  gateways handle it. ISO-TP is the integration point for future
  raw-CAN transports, not for the current production stack.
- **The KWP2000 timeout fix is Tier A + Tier B because it touches
  `src-tauri/src/transport/kdcan.rs`** — protected path. Single
  small Tier B change. The default deadline raises from 1000ms to
  3000ms for known-slow modules (CIC, CAS) per a per-target
  override table.
- **E-series broadcast decoder is Tier A frontend only.** The
  transport already exposes the bytes via `TrafficEntry`; the
  renderer just needs to subscribe and parse.
- **No new crate deps.** The KWP change is `std::time::Duration`
  only; the decoder is pure data → numbers.
- **No `protocol/` changes.** UDS/KWP builders don't move; we're
  changing the request deadline and adding a new frame parser.

## What we will NOT do (this cycle)

- **Wire ISO-TP into production.** Correctly not wired; the gateways
  terminate. The `isotp.rs` module stays as the integration point
  for future raw-CAN transports (SocketCAN, OBDLink STN, future
  DoIP socket).
- **Re-implement ISO-TP.** Same as v0.12.0; 10 existing unit tests
  cover it.
- **Add a SocketCAN or OBDLink STN transport.** That's a v0.14.0+
  follow-up that *would* wire ISO-TP into production, because
  those transports *do* need the multi-frame reassembly. Today
  they don't exist.
- **Full F-series DTC dump coverage.** Multi-frame reassembly is
  the gating dep; without ISO-TP-wired transports, the production
  app still truncates single-frame UDS responses. The broadcast
  decoder gives users a way to *see* live data on E-series
  without that dependency. A v0.14.0+ raw-CAN transport would
  unlock the rest.
- **Touch `enet.rs::request`.** The 1000ms deadline on ENET is
  fine — ENET's RTT is sub-100ms on a wired connection. The KWP
  problem is specifically K+DCAN serial.
- **Cross-cutting connect() refactors.** Touching one transport's
  request deadline is fine; rewriting the request signature is
  not.

  it and the old 1000ms must reject it.
- Frontend: no UI change needed. The behavior is invisible until a

  now succeeds.

## Premise

The ISO-TP multi-frame reassembly (`src-tauri/src/transport/isotp.rs`,
430 lines + 10 unit tests) has been **implemented and tested since v0.6.0**
but is **not wired into the live transport stack** — the production
`connect()` path in `kdcan.rs` and `enet.rs` constructs bare
`KdcanTransport`/`EnetTransport`, not `IsoTpTransport`. So:

- Long UDS responses (real F/G VIN, full DTC dumps) silently truncate
  to a single CAN frame's 7 bytes.
- E-series CAS multi-frame responses look like garbled bytes when
  the VIN really IS there.
- The "ISO-TP multi-frame" item has been marked 🟢 Ready in the
  ROADMAP for cycles, because the work to **wire it up** is what
  v0.13.0 lands — not the implementation (already done).

Adjacent 🟢 Ready item: **KWP2000 slow-module timeout fix** — the
hardcoded 1000ms deadline in `kdcan.rs::request` times out on real
CIC/CAS modules. Small, well-scoped fix.

And: E-series CAN broadcast frame decoding (0x0AA RPM, 0x1D0
coolant, 0x545 oil temp) — the bytes are already on the bus from
the transport; the renderer just doesn't decode them. Pure
frontend.

## Honest ceiling

- **Wire, don't re-implement.** ISO-TP is done; v0.13.0 plugs it in.
  Any new implementation in `isotp.rs` is a red flag — that work is
  tested and locked.
- **Tier B for the wire-up only.** The actual change is small
  (two `connect()` paths switch from raw `Transport` to
  `IsoTpTransport<RawBus>`); the file count is low but the
  protected-path exposure is real. Single Tier B PR, gated behind
  human merge per CLAUDE.md §1.
- **No new crate deps.** `isotp.rs` is stdlib + tokio, same as the
  rest of the transport layer.
- **No protocol changes.** UDS/KWP request builders don't move;
  the reassembly is invisible to the protocol layer (it sits between
  the wire and the existing `Transport` trait).
- **No regressions for short responses.** Single-frame messages
  pass through `IsoTpTransport` unchanged — the existing tests
  in `isotp.rs::tests` cover this.

## What we will NOT do (this cycle)

- **Re-implement ISO-TP.** Don't touch `isotp.rs` unless a real bug
  surfaces; the tests already cover FF/CF/FC paths and CAS fallback.
- **DoIP/ENET multi-segment changes.** ENET already calls
  `IsoTpTransport::with_timeouts(...)` for its inner transport;
  any ENET-specific changes belong in a follow-up.
- **Blocker tests for the wiring.** Slice 2's wire-up is small
  enough that the existing 10 isotp tests + a 3-line `connect`
  integration test covers the risk.
- **New ISO-TP features** (e.g. extended addressing, mixed-addressing,
  TP 2.0). Out of scope; the wire-up uses TP 1.0 which is what every
  BMW diagnostic session uses.
- **Cross-cutting refactors.** Touching `connect()` in two places
  is fine; rewriting the connect signature is not.

## Tier split
