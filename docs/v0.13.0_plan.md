*Drafted 2026-07-22 against `main` @ 7dd7fd1 (v0.12.0 "Fault Memory" closed; #143ŌĆō#149 merged).*

## Premise

The ISO-TP multi-frame reassembly (`src-tauri/src/transport/isotp.rs`,
430 lines + 10 unit tests) has been **implemented and tested since v0.6.0**
but is **not wired into the live transport stack** ŌĆö the production
`connect()` path in `kdcan.rs` and `enet.rs` constructs bare
`KdcanTransport`/`EnetTransport`, not `IsoTpTransport`. So:

- Long UDS responses (real F/G VIN, full DTC dumps) silently truncate
  to a single CAN frame's 7 bytes.
- E-series CAS multi-frame responses look like garbled bytes when
  the VIN really IS there.
- The "ISO-TP multi-frame" item has been marked ­¤¤ó Ready in the
  ROADMAP for cycles, because the work to **wire it up** is what
  v0.13.0 lands ŌĆö not the implementation (already done).

Adjacent ­¤¤ó Ready item: **KWP2000 slow-module timeout fix** ŌĆö the
hardcoded 1000ms deadline in `kdcan.rs::request` times out on real
CIC/CAS modules. Small, well-scoped fix.

And: E-series CAN broadcast frame decoding (0x0AA RPM, 0x1D0
coolant, 0x545 oil temp) ŌĆö the bytes are already on the bus from
the transport; the renderer just doesn't decode them. Pure
frontend.

## Honest ceiling

- **Wire, don't re-implement.** ISO-TP is done; v0.13.0 plugs it in.
  Any new implementation in `isotp.rs` is a red flag ŌĆö that work is
  tested and locked.
- **Tier B for the wire-up only.** The actual change is small
  (two `connect()` paths switch from raw `Transport` to
  `IsoTpTransport<RawBus>`); the file count is low but the
  protected-path exposure is real. Single Tier B PR, gated behind
  human merge per CLAUDE.md ┬¦1.
- **No new crate deps.** `isotp.rs` is stdlib + tokio, same as the
  rest of the transport layer.
- **No protocol changes.** UDS/KWP request builders don't move;
  the reassembly is invisible to the protocol layer (it sits between
  the wire and the existing `Transport` trait).
- **No regressions for short responses.** Single-frame messages
  pass through `IsoTpTransport` unchanged ŌĆö the existing tests
  in `isotp.rs::tests` cover this.

## What we will NOT do (this cycle)

- **Re-implement ISO-TP.** Don't touch `isotp.rs` unless a real bug
  surfaces; the tests already cover FF/CF/FC paths and CAS fallback.
- **DoIP/ENET multi-segment changes.** ENET already calls
  `IsoTpTransport::with_timeouts(...)` for its inner transport;
  any ENET-specific changes belong in a follow-up.
- **Blocker tests for the wiring.** Slice 2's wire-up is small
  enough that the existing 10 isotp tests + a 3-line `connect`
  integration test covers the risk.
- **New ISO-TP features** (e.g. extended addressing, mixed-addressing,
  TP 2.0). Out of scope; the wire-up uses TP 1.0 which is what every
  BMW diagnostic session uses.
- **Cross-cutting refactors.** Touching `connect()` in two places
  is fine; rewriting the connect signature is not.

## Tier split

- **Tier B (1 PR):** Wire `IsoTpTransport` into `kdcan.rs::connect()`
  and `enet.rs::connect()`. Touches `src-tauri/src/transport/{kdcan,enet}.rs`.
- **Tier A (4 PRs):**
  - Frontend integration test for VIN via ISO-TP (uses the existing
    simulator `vin_multi_frame` personality ŌĆö `tests/transport/isotp.rs`
    has the helpers).
  - KWP2000 slow-module timeout fix ŌĆö configurable deadline + per-target
    override table; default raised for known-slow modules.
  - E-series CAN broadcast frame decoder ŌĆö pure frontend, parses the
    already-broadcast bytes into a Live Gauges panel.
  - `docs/validation/multi-frame.md` ŌĆö what changed and how to verify.

### Slice 2 ŌĆö Wire `IsoTpTransport` into the production transport stack (Tier B, Rust)

The actual change:

- `src-tauri/src/transport/kdcan.rs::connect(...)` currently returns
  a bare `KdcanTransport`. Change the return shape (or wrap it) so
  the trait-object returned is `Box<dyn Transport>` over an
  `IsoTpTransport<KdcanCanBus>` instead of `KdcanTransport`.
- Same for `enet.rs`. The ENET path already constructs an
  `IsoTpTransport` internally ŌĆö verify the wiring and surface the
  result.
- The simulator (`sim.rs`) already exposes `SimCanBus: isotp::CanBus`,
  so the `connect()` for the simulator path also gets the upgrade.
- Test: extend `tests/async_commands.rs` to assert the connected
  transport goes through `IsoTpTransport`. (The existing 10
  `isotp.rs` unit tests cover the reassembly logic; this is a
  smoke test that the wiring is in place.)
- Update `src-tauri/src/transport/mod.rs` docs to reflect the
  upgraded transport shape.

Single Tier B PR. Flag `src-tauri/src/transport/{kdcan,enet}.rs`
at the top of the body, wait for human merge.

### Slice 3 ŌĆö VIN-via-ISO-TP regression test (Tier A, frontend test)

Pure verification ŌĆö proves the wire-up actually works end-to-end

- A new test in `src/js/test/` (or extend an existing one) that

### Slice 4 ŌĆö KWP2000 slow-module timeout fix (Tier A, frontend + Rust)

- `kdcan.rs::request` hardcodes 1000ms. Replace with a configurable
  per-target deadline ŌĆö default 1000ms, but CIC (0x01), CAS (0x40),
  and other known-slow modules get 3000ms by default.
- Configurable via a new constructor parameter on `KdcanTransport`,
  with the existing `KdcanTransport::open(...)` keeping the current
  defaults (so callers don't need to change).
- Frontend: no UI change needed. The behavior is invisible until a
  user reads faults on an E-series CIC ŌĆö what was a TimeoutError
  now succeeds.
- Tests: a scripted transport (like the one in `isotp.rs::tests`)
  that delays its response by 2500ms; the new deadline must accept
  it and the old 1000ms must reject it.

This is the **secondary** value of v0.13.0 ŌĆö a small bug fix that
unblocks more real-car work than ISO-TP does, because every E-series
owner hits it on their first fault read.

### Slice 5 ŌĆö E-series CAN broadcast frame decoder (Tier A, frontend)

The CAN broadcast frames are already on the bus; the transport
records them in `TrafficEntry`; the renderer just doesn't decode
them. This slice adds a Live Gauges panel that:

- Subscribes to the existing traffic log filtered to the broadcast
  IDs (0x0AA, 0x1D0, 0x545).
- Decodes the 2-byte payload of each into a gauge (RPM, coolant ┬░C,
  oil temp ┬░C).
- Renders as a new "Live Gauges" tab or side panel ŌĆö same Tier A
  surface as the v0.4.0 Live Data tab.
- Uses the same `Gauge.set` canvas widget from `src/js/gauges.js` ŌĆö
  no new rendering code, just wiring.
- Tests: pure-data decode functions for each ID (no DOM), with
  fixtures for known-good and known-bad inputs.

Bonus: for E46 owners, this gives them **a working tachometer** that
the current app doesn't show. For F/G owners, this is the second-best
path to live data (DME PID 0x0C is still the canonical RPM source).

## Why "Real Reads, Real Long" as the cycle name

Cycle names track the user promise. v0.11.0 made sessions shareable.
v0.12.0 made the app remember. **v0.13.0 makes real reads work** ŌĆö the
ones that have been silently truncated or timing out since v0.3.0.
"Real long" reads because the wire-up is what unlocks full F/G DTC
dumps (12+ bytes per entry, 10+ entries ŌåÆ 200+ byte responses that
currently truncate to one frame).
