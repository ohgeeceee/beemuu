# BeeEmUu v0.5.0 — Cycle Plan

> **Status:** v0.4.0 cut on 2026-07-15 (PR #69). v0.5.0 planning
> starts here. The cycle scope below is the **proposed** spine
> — open a Discussion thread to argue with it before the first
> PR lands, per `COMMUNITY_FRAMEWORK.md`.

## Premise

v0.4.0 finished the decoder + UI plumbing for tuner-style work:
every byte from the bus can now be classified (numeric decoded
or enum-labelled), the histogram viewer turns the log into
distributions, and the service-functions layer is ready for
chassis-validated routine IDs. What's missing isn't more
plumbing — it's **real-car evidence** and the small features
that depend on it.

v0.5.0 picks the cycle name **"Ground Truth"** because the work
is about validating the abstractions we shipped in v0.3 / v0.4
against real hardware, and adding the narrow features that
real-car owners actually need first.

## Cycle scope (proposed)

**Three small PRs**, in order. Each is small enough to ship on
its own, each doc-only-or-protected as appropriate.

### PR #1 — Real-car u8_enum validation harness

**Files**

- New `docs/validation/u8_enum-validation.md` — a checklist
  for an F/G-series owner with an ENET adapter to validate
  the example enum DIDs (`gear` / `engine_state` /
  `knock_detect`) in `community/profiles/{b58,n55}.toml`.
  Each DID gets a small table: byte → expected state,
  pass/fail checkbox, screenshot upload guidance.
- New `community/profiles/b58.toml` comment block pointing at
  the validation doc.
- No code change. Doc-only — auto-merge eligible.

**Why first.** The example DIDs were marked `[needs verification]`
in PR #60. Until they're validated on a real car, the u8_enum
foundation is theoretical. A validation harness gives us the
smallest possible next step (a doc that someone with a car
fills in) without blocking the rest of the cycle on real-car
access.

**Acceptance.** Doc lands; an F/G-series owner can pick it up
without further help.

### PR #2 — Real-car fuel-trim / adaptation readout

**Files**

- `community/profiles/{b58,n55}.toml` — add 2–3 DIDs for
  short-term and long-term fuel trims (typically DIDs `0x4116`,
  `0x4117`, `0x4118` on BMW DME; community-sourced). Mark
  `[needs verification]` until validated.
- `src-tauri/src/data/live.rs` — likely no new decoders needed
  (existing `s16_div100` handles the typical scaling). If a
  multiplier differs, add it.
- `community.rs` — no shape change (TOML loader already handles
  arbitrary DID rows).
- Protected path (`src-tauri/src/data/live.rs` is adjacent to
  `protocol/`). Flag for human review.

**Why second.** Tuners cite fuel-trim data as the #1 thing
they want to see; it was on the v0.4 🟡 list ("Adaptation /
fuel trim readout"). The decoder foundation from v0.4 is ready
to surface it; only the DIDs + verification are missing.

**Acceptance.** Profile TOMLs parse, live-data tick returns
non-NaN numbers for the new DIDs in the simulator. Real-car
numbers still `[needs verification]`.

### PR #3 — Real-car knock-detection visualisation polish

**Files**

- `src/js/main.js` — extend the existing Logging tab's
  per-channel display to flag a `knock_detect` channel
  (or any enum-style channel with severity levels) with a
  visual indicator when the value isn't `0` / "None". Uses the
  `LiveValue.text` field from PR #60 / #64.
- `src/css/app.css` — small style addition for the indicator.
- `src/js/live_format.js` — small helper to decide "is this a
  severity-bearing enum?" (testable).
- Protected path (`src/js/` is JS-only — no protected Rust).
  Flag for human review per the project's PR discipline.

**Why third.** Knock detection was on the v0.4 🟡 list. The
u8_enum decoder shipped in v0.4 means the frontend now has
text labels ("None" / "Light" / "Moderate" / "Severe") for
the knock state. The only missing piece is visual emphasis
on non-zero states — the natural next step.

**Acceptance.** When `LiveValue.text === "Moderate"` or
`"Severe"`, the gauge/chart shows a visible marker. Verified
against simulator data; real-car validation pending.

## What does NOT ship in v0.5.0 (deferred)

These stay on the v0.5.0+ 🟡 list per
[`ROADMAP.md`](../../ROADMAP.md) — they're real work but
not part of this cycle's spine:

- **OBDLink MX+ support** — needs adapter on hand + BLE stack
  work. Likely a v0.6+ cycle.
- **ENET/DoIP auto-detection** — needs protocol state-machine
  work and real-car testing across multiple chassis. v0.6+.
- **Trigger-based logging** — needs a trigger engine and
  state machine; design lives in `docs/AGENTS_SETUP.md` as a
  future research item.
- **Log file merge / comparison** — defer to v0.6+ until the
  histogram work has been validated in production. Spec lives
  in the v0.4 ROADMAP.
- **Custom math channels** — needs a safe expression sandbox;
  research-grade work.

These stay on the Backlog (separate section in
[`ROADMAP.md`](../../ROADMAP.md)):

- Cloud sync (opt-in log upload) — privacy + ops story first
- Raspberry Pi CAN bridge — hardware project of its own
- Plugin system for custom decoders — community governance work
- Bootmod3 / MHD integration — legal risk
- Multi-language UI — translation coordination problem
- Web-based shared-log viewer — needs hosted backend work

## Risks and tradeoffs

- **Sibling-agent race** — same multi-writer pattern as v0.4.
  Always work in a fresh sibling clone, branch off `origin/main`,
  push from the clone.
- **`origin/main` drift** — local `main` was stale at `e28914b`
  while `origin/main` was at `0e5e0e2` during the v0.4 cycle.
  Always `git fetch origin` before basing a branch, and diff
  against `origin/main`, not `main`.
- **CLAUDE.md rule 2** — only doc-only PRs auto-merge. PRs #2
  and #3 must go through human review; do not bypass.
- **Real-car evidence** — nothing in v0.5.0 should claim
  "verified on a real N55/B58" unless someone has actually
  validated it on a real car. The `[needs verification]`
  markers stay until then.

## Test plan per PR

- **PR #1** — no automated tests. Doc-only; manual checklist
  fills itself in as a real owner completes it.
- **PR #2** — `cargo test --lib` (extend `live` tests if a
  new decoder is added; otherwise existing tests cover it).
  Profile TOML parses via `tomllib`.
- **PR #3** — `npm run test:js` (extend `live_format.test.cjs`
  if a helper is added; otherwise existing tests cover it).
  Manual simulator verification.
