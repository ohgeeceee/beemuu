# v0.8.0 Plan — "Service Bay"

*Draft 2026-07-16, after v0.7.0 "Unblockers" merged (PRs #108 / #109 /
#110). This plan landed as a PR directly; review happened in the PR
per `COMMUNITY_FRAMEWORK.md` Rule 2.*

## Premise

Seven release cycles built the diagnostic spine: read fault memory,
decode live data, log it, compare it, and connect over K+DCAN **or**
ENET/DoIP without touching an IP address. What an owner still cannot
do is the thing ISTA+ exists for in a workshop: **run the service
functions** (resets, adaptations, bleeding, registrations) and **cover
the whole car** (every module, every common fault text, every
mainstream engine).

The gap analysis below says the highest-value shippable work is
exactly that: service-function breadth and coverage breadth (fault
texts, ECU scan table, engine profiles) — plus fixing two data files
that shipped **truncated** and currently contribute nothing.

## Milestone theme

**"Service Bay"** — turn the diagnostic reader into the service
workstation: more service functions (with honest verification status),
more of the car visible (scan table, fault texts), more engines
covered (profiles), and the data-integrity floor that keeps all of it
trustworthy.

## PR-sized slices

Four PRs, mirroring v0.7.0's shape. Tier per `CLAUDE.md`.

### PR #1 — Data integrity: fault-text rescue, corpus growth, TOML parse gate (Tier A)

**Why first.** Two shipped files are broken today
(`community/dtc_texts.toml` truncated → parse error → zero overlay
entries; `CONTRIBUTING.md` truncated mid-table), and **CI cannot catch
this class of breakage** because `lint-toml.js` checks whitespace, not
TOML validity. Everything else in this cycle adds more data files; the
gate must exist before the pile grows.

**Files**

- `community/dtc_texts.toml` — repair the truncated tail; grow the
  table from **in-repo grounded sources only**:
  `research/bmw_diag_dim01_dtcs.md` and the curated seeds in
  `backend/seed_dtcs.py` / `backend/seed_bmw_dim01.py` (~450 rows the
  project already vetted). Every entry keeps a source comment; no
  ISTA-extracted text (community rule).
- `src-tauri/src/community.rs` — add a unit test that parses **every
  shipped community TOML** (`dtc_texts.toml`, `freeze_schemas.toml`,
  `profiles.toml`, `profiles/*.toml`) so a future truncation fails
  `cargo test`, which CI already runs (`test.yml`). No production-code
  change expected beyond test-only visibility helpers.
- `CONTRIBUTING.md` — complete the truncated file: finish the
  confidence-label table, add the promised Parameter Explorer / code
  contribution / development-setup sections, and document the
  **verification-label conventions** in one place (see "Docs/community"
  below).
- `CHANGELOG.md` — bullets for the rescue + corpus + gate.

**Acceptance.** `python -c "import tomllib"` parses every
`community/**/*.toml`. The new cargo test fails when a fixture TOML is
deliberately broken (proven in the PR by a temporary bad fixture, then
removed). `node scripts/lint-toml.js` clean. Fault-text count reported
before/after in the PR body.

### PR #2 — Service-function breadth with honest verification status (Tier B)

**Why second.** Six simulator-grade routines is a demo, not a service
bay. But write-path routine IDs must never be invented — the v0.5.0
`[needs verification]` discipline for *reads* becomes a hard
`[UNVERIFIED]` gate for *writes*.

**Protected paths:** `src-tauri/src/data/service_functions.rs`,
`src-tauri/src/commands.rs` (command surface), possibly
`src-tauri/src/protocol/` (routine result handling). Flag at the top
of the PR; human merge.

**Files**

- Audit first: `research/bmw_diag_dim04_uds_dids.md`,
  `TECH_SPECS.md`, and the simulator's routine table are the only
  admissible sources. The PR body lists each candidate routine ID and
  its source; unsourced candidates ship in a documented "known-missing"
  list instead (DPF regen, throttle/valvetronic adaptation, steering
  angle calibration, EGS adaptation reset, EMF service mode).
- `src-tauri/src/data/service_functions.rs` — add a verification
  marker to `ServiceFunction` (or encode it in the label, matching the
  profile-label convention); add sourced entries only. The existing
  count/ID-preservation tests are updated **deliberately and called
  out in the PR body** — that test is a contract, and evolving a
  contract is a conscious act, not a drive-by.
- `docs/validation/service-functions.md` (new) — real-car validation
  harness mirroring `u8_enum-validation.md` / `injector-validation.md`:
  preconditions (battery maintainer, engine state), what to observe,
  how to report. This doc is what converts future "works on my F30"
  comments into `[verified]` upgrades.
- `src/index.html` / `src/js/main.js` — render the verification
  marker; `[UNVERIFIED]` entries keep the existing risk confirmation
  and gain a second "routine ID not chassis-validated" line.
- `CHANGELOG.md`.

**Acceptance.** Every new entry names its routine-ID source in a
comment; entries without a source are absent (listed in the doc
instead). `cargo test` green including the deliberately-updated
contract tests. UI shows the verification marker on all six existing
entries (they are simulator-grade → `[UNVERIFIED]` by definition).
Real-car testing is **not** a merge blocker; the harness doc is the
path to it.

### PR #4 — ECU scan-table breadth + addressing-model doc (Tier A)

**Why fourth.** The 12-address scan table is E-series-era. F/G cars

**Files**

- Audit: `research/bmw_diag_dim0*.md`, `TECH_SPECS.md`, simulator
  module table. Each candidate address needs an in-repo citation.
- `src-tauri/src/data/ecus.rs` — sourced additions only (read-only
  probing → not a write path → Tier A). Update the scan-table test.
- `src-tauri/src/transport/sim.rs` — simulator answers the new

- `TECH_SPECS.md` or `docs/hardware/README.md` — short addressing-model

**Acceptance.** `cargo test` green; simulator scan shows the new

## What we will NOT do (and why)

Honesty is a feature. These ISTA+ capabilities stay out of BeeEmUu —
permanently unless stated otherwise:

- **Programming / flashing.** A failed flash bricks the module and
  BeeEmUu has no recovery path. It also needs SVT/CAFD data that ships
  only inside BMW's proprietary stack, plus session-stability
  guarantees (30+ minute programming sessions, voltage maintenance)
  this codebase does not claim. **Permanent exclusion.**
- **FSC certificates / AOS activation.** Bound to BMW's backend;
  generating or sideloading certs circumvents BMW's licensing —
  legally fraught for the project and for users. **Permanent
  exclusion.**
- **ECU coding writes (NCD/CAFD).** Out of v0.8.0 scope: the checksum
  and CAFD container formats are undocumented in-repo, and a bad write
  is a bricked module. Read-only coding display may become a research
  item once someone documents the format from open sources; writing
  stays Tier C until that story exists.
- **ISTA repair-instruction / wiring-diagram corpus.** Copyrighted
  BMW material; redistributing it would endanger the project.
  Community-sourced schematics (the existing `schematics.rs` path)
  are the ceiling.
- **Immobiliser / key learning / CAS functions.** Theft-enabling;
  no owner use case outweighs it. **Permanent exclusion.**
- **Bootmod3 / MHD integration.** Legal risk (their ToS and BMW's);
  already deferred in v0.5.0/v0.6.0 — stays deferred.
- **Cloud sync of logs.** Needs a privacy + ops story first; the
  anonymised snapshot-share path covers the legitimate sharing case.

## Risks and tradeoffs

- **Write-path temptation.** PR #2 is where a well-meaning
  contributor pastes forum routine IDs. The audit-first rule (in-repo
  source or it doesn't ship) is the whole design; the PR template
  should name the source per entry.
- **Contract-test edits.** PR #2 deliberately updates the
  service-function preservation tests. The PR body must call this out
  explicitly; silent contract edits are how regressions sneak in.
- **Scan-time cost.** PR #4's wrong-address cost is paid by users as
  timeouts. Sourced-only additions; when in doubt, the address goes in
  the doc's "unknown" list, not the table.
- **CLAUDE.md staleness.** This plan flags it but cannot fix it —
  CLAUDE.md edits are Tier C. Owner action requested: mark keep-alive
  (issue #87), ISO-TP (#88), and `read_vin` as shipped.
- **`origin/main` drift** — same discipline as prior cycles: fetch and
  branch off `origin/main`, never local `main`.
- **GH007 private-email push block** — author commits as
  `ohgeeceee@users.noreply.github.com` from the start.
- **Direct-PR model** — per `COMMUNITY_FRAMEWORK.md` Rule 2, this plan
  and its slices land as PRs directly; no Discussion thread gates them.

## Why "Service Bay" as the cycle name

v0.4 → v0.7 names tracked the *user promise*: tune it, ground it in
real hardware, then remove friction. v0.8.0's promise is the workshop
workflow itself — the moment an owner finishes a brake job and reaches
for BeeEmUu instead of a $200 scan tool or a $1,500 ISTA subscription.
The name is honest about scope: a service bay does resets,
registrations, and bleeding — it does not flash ECUs, and neither do
we.
