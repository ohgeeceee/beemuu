# BeeEmUu v0.5.0 — Real-car u8_enum Validation Harness

> **Status:** v0.5.0 cycle started 2026-07-15 (PR #69 cut v0.4.0;
> PR #70 published the cycle plan). This is the first PR of the
> v0.5.0 "Ground Truth" cycle. **Doc-only. Auto-merge eligible.**

## Why this PR exists

PR #60 (v0.4.0) shipped the `u8_enum` decoder + per-parameter
enum-map pipeline. It added example enum DIDs to
`community/profiles/{b58,n55}.toml`:

- `gear` (DID `DA0A`, EGS)
- `engine_state` (DID `4004`, DME)
- `knock_detect` (DID `401F`, DME)

Each carries `[needs verification]` markers in the label because
the byte-to-state mappings came from OBDb (CC-BY-SA 4.0) and have
not been validated against a real car.

Until an F/G-series owner with an ENET adapter runs the binary
against their car and confirms the mappings, the foundation is
theoretical. **This PR gives that owner a small checklist they
can fill in without further help from a developer.**

## What this PR ships

A new doc, **`docs/validation/u8_enum-validation.md`**, that
contains:

1. **Pre-flight checklist** — what to check before running the
   tests (cable + adapter, ignition on, profile selected,
   simulator NOT in use).
2. **Per-DID test procedure** — three identical-shape tables,
   one per DID (`gear`, `engine_state`, `knock_detect`), each
   covering:
   - DID address + query string
   - Expected states per byte value (from OBDb)
   - Pass/fail checkbox per byte
   - Notes / screenshot upload guidance
3. **Results submission instructions** — how to file the
   completed checklist (PR against `community/profiles/b58.toml`
   / `n55.toml`, removing the `[needs verification]` markers and
   adding `[verified 2026-MM-DD on chassis XYZ]` to the label).
4. **What to do if a byte doesn't match** — log it, don't fake a
   pass, open an issue with the byte value and chassis.

Also: a new `docs/v0.5.0_first_pr.md` (mirrors
`docs/v0.4.0_first_pr.md`'s shape) — the written record of why
this PR exists and the cycle-spine context.

## What this PR does NOT do

- **No code change.** Doc-only.
- **No new tests.** The validation is manual — a person running
  the app against a real car. Automation of "did this state
  actually fire on a real engine" requires either a hardware
  bench rig or a much larger simulator investment. Both are
  out of scope for v0.5.0.
- **No change to the `[needs verification]` markers.** Those
  stay until an F/G owner files the results PR.

## Scope-discipline notes

- This PR is the **first PR of v0.5.0**. It can land without
  any of the other v0.5.0 PRs being ready. The cycle plan
  ([`docs/v0.5.0_plan.md`](../v0.5.0_plan.md)) lists three
  PRs in order; PR #1 lands first because it's the smallest
  possible next step and unblocks the rest.
- Real-car validation is the only 🟡 blocker for full
  v0.4.0-era completion (the original "Tuner Friendly" cycle
  was gated on real-car data for the example DIDs). This PR
  is the path to retiring that blocker.
