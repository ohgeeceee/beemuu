# BeeEmUu v0.7.0 — Cycle Plan

> **Status:** v0.6.0 cycle closed on 2026-07-16 (PR #82 cuts
> the release; pending human review at time of authoring).

> feature without a Discussion" rule, **no v0.7.0 work

## Premise

v0.6.0 finished the "Real Hardware" cycle: log-merge
comparison, injector-time validation harness, and OBD-II
mode 01 PID auto-discovery. The pattern across v0.4 →
v0.5 → v0.6 has been **3-PR spine cycles, each narrow,
each well-bounded**. What's left isn't more spine work —
it's the **Ready-to-Claim 🟢 pile** that has accumulated
over three cycles, plus one piece of high-leverage protocol
work that unblocks multiple downstream 🟡 items.

v0.7.0 picks the cycle name **"Unblockers"** because the
work is about removing the small UX / data-shape /
protocol blockers that have been sitting on the ROADMAP
since v0.5.0 (consolidated in PR #79). Each PR is
independent and ships without depending on the others.
A first-time contributor can land any single PR and earn
their first merged commit on the project.

This is the **narrowest defensible scope** consistent with
the 3-PR spine shape of v0.4.0 / v0.5.0 / v0.6.0. Each
item is small, well-bounded, and ships independently.
Nothing in this plan is a blocker for any other item.

## Cycle scope (proposed)

**Three small PRs**, in order. Each is small enough to ship
on its own, each pure-data / doc-only / non-protected as
appropriate. PR #3 (ENET/DoIP auto-detection) is the only
one that touches a protected path; the others are doc-only
or pure profile data.

### PR #1 — ENET/DoIP auto-detection

**Why first.** ENET/DoIP auto-detection is the
**highest-leverage protocol work on the entire ROADMAP**.
Multiple 🟡 items in the v0.6.0 needs-research list
(Real-car validation B58 / N55 F-series, OBDLink MX+
support, full-distribution knock visualisation, etc.)
become tractable once a contributor can plug in an ENET
cable and have the app discover the car's VIN/IP
without typing `169.254.x.x` into a text box. CLAUDE.md
already calls this out as an invariant: *"F/G-series
uses DoIP: broadcast UDP discovery to port `13400` and
use the VIN/IP the car returns (typically `169.254.x.x`).
Broadcast across all active interfaces; never hardcode
a target IP."* The hardcoded-IP shape in
[`src/index.html`](src/index.html) (the
`<input id="conn-addr" value="169.254.16.11:6801">` default)
is the *only* remaining hardcoded car IP in the app.

**Files**

- `src-tauri/src/transport/enet.rs` — new UDP broadcast
  listener on port `13400` (SAE J1979 DoIP vehicle
  identification). Listen for the car's VIN/IP response,
  parse the activation code, and return a list of
  discovered ECUs.
- `src-tauri/src/transport/mod.rs` — extend
  `TransportConfig` with an `auto_discover: bool` flag;
  when set, the `open()` call broadcasts first and uses
  the discovered IP.
- `src-tauri/src/commands.rs` — new
  `#[tauri::command] discover_enet_targets() ->
  Result<Vec<EcuInfo>, String>` for the UI to surface
  the discovered list before connecting.
- `src-tauri/src/lib.rs` — register the new command.
- `src-tauri/src/protocol/mod.rs` — possibly extend
  `identify()` with a UDS `22 F1 90` (VIN) variant for
  ENET-discovered modules. Existing `read_vin` already
  handles this case; verify the ENET path uses it.
- `src/index.html` + `src/js/main.js` — UI: a
  "Discover" button next to the manual `conn-addr` input.
  When the user picks a discovered target, the manual
  field auto-fills with the discovered IP. Falls back to
  the manual IP if discovery times out (5-second window
  is plenty for a real-car broadcast on a quiet network).
- `src/css/app.css` — small style block for the discovery
  results panel.
- New unit tests for the broadcast parsing in
  `transport/enet.rs`.

**Protected path**: `src-tauri/src/transport/**` is on
the CLAUDE.md Protected Paths list. Flag at the top of
the PR description per project discipline. **Not
auto-merge eligible.**

**Why second-priority despite being most-leverage.** The
ENET transport currently has **no unit-test surface for
the broadcast path** — UDP sockets are hard to mock.
The existing `EnetTransport::request()` is
straightforward, but the discovery loop is new code on
top of a transport that the project's CI doesn't
actually exercise end-to-end (no real cars in CI). The
right move is **test the broadcast parser against a
scripted UDP response in the same `ScriptedTransport`
pattern** as the v0.6.0 PR #81 OBD-II PID scan, then
verify the activation-handshake bytes against the spec.

**Acceptance.** User clicks "Discover," sees a list of
ECUs (DME, EGS, CAS, etc.) with their VIN and IP within
~5 seconds. Manual IP entry still works as a fallback
(some adapters don't broadcast — keep the escape hatch).
`cargo test --lib` passes including the new broadcast
parser tests. The manual-IP default in `src/index.html`
becomes a *placeholder* instead of a hardcoded address.

### PR #2 — Ready-to-Claim pile: dark/light theme + save/load workspace layout + gauge theming (S55 colors)

**Why second.** The 🟢 Ready-to-Claim pile (consolidated
in PR #79, in turn deriving from ROADMAP v0.3.0's
"Ready" section) has been sitting since v0.4.0. Three
items in particular are **all small, all UI-only, all
on the ROADMAP but never claimed**:

1. **Dark/light theme toggle** (`src/css/app.css` — CSS
   variables exist) + toggle button in `src/index.html`.
   The dark theme is the only theme today; the CSS
   variable layer is already in place for light. UI only,
   no Rust touched. ~30–60 lines of CSS + a small JS
   handler that flips `body[data-theme]`.

2. **Save/load workspace layout** — remember which gauges
   the user had open and at what min/max bounds. Mid-sized
   but pure JS, no backend. Uses Tauri settings persistence
   or a `~/beeemuu/workspace.json` file. ~150–200 lines.

3. **Gauge theming (M colors for S55)** — per-profile color
   schemes. The example profile TOML format already
   supports a `[theme]` block; the gauge canvas just needs
   to read it. ~50 lines of JS + a small `[theme]` block in
   `community/profiles/s55.toml` (which doesn't exist yet —
   see PR #3 below; this PR ships the gauge-side code, PR
   #3's S55 profile uses it).

**Files**

- `src/css/app.css` — light-theme CSS variables (already
  declared for some elements; complete the sweep).
- `src/index.html` — theme toggle button in the header.
- `src/js/main.js` — `toggleTheme()` handler; reads/writes
  `localStorage.beeemuu.theme` (or Tauri settings if
  preferred); flips `body[data-theme]`.
- `src/js/workspace.js` (new) — pure helper, `save()`
  and `load()` for the gauge layout. JSON shape:
  `{gauges: [{profile_id, param_id, min, max}, ...]}`.
- `src/js/test/workspace.test.cjs` (new) — round-trip
  save/load tests.
- `src/js/gauges.js` — apply per-profile `[theme]` block
  colors; fall back to current blue/green defaults if no
  theme block.
- `docs/DECODE_FUNCTIONS.md` § 9 (new) — per-profile
  `[theme]` block syntax.
- `CHANGELOG.md` — one bullet per sub-feature.

**Doc-only / pure JS** — **auto-merge eligible** per
CLAUDE.md rule 2.

**Acceptance.** Toggle works without reload. Workspace
layout persists across app restarts. `node --test
src/js/test/workspace.test.cjs` passes; no regressions
in existing tests.

### PR #3 — Engine profile gap fills: N20 / N26 + S55 TOMLs

**Why third.** The v0.6.0 ROADMAP and the Ready-to-Claim
pile both list N20/N26 and S55 engine profiles as small
new TOML files. They're the **only two mainstream BMW
engines missing** from the example profile set:

| Engine | Profile present? | Notes |
|---|---|---|
| B58 | ✅ `b58.toml` | V0.3.0 base; refined in v0.4/v0.5 |
| N55 | ✅ `n55.toml` | V0.3.0 base; fuel-trim in v0.5 |
| N52 | ✅ `n52.toml` | E-series, KWP2000 |
| N54 | ✅ `n54.toml` | E-series twin-turbo, KWP2000 |
| N62 | ✅ `n62.toml` | V8, E-series |
| N20 / N26 | ❌ | F-series 4-cyl, modular engine family |
| S55 | ❌ | F-series M3/M4 high-perf N55 variant |

Each new profile is a **straightforward clone of
`b58.toml`** with engine-specific DIDs (sourced from
`TECH_SPECS.md`, not forum threads) and the
`[needs verification]` marker on every entry until
real-car validation. No Rust change. No protocol
change. No JS change.

**Files**

- `community/profiles/n20.toml` (new) — clone `b58.toml`,
  swap DIDs for N20-specific ones. All entries marked
  `[needs verification]`. ~200 lines.
- `community/profiles/s55.toml` (new) — clone `n55.toml`,
  add S55-specific high-perf DIDs (higher HPFP target,
  oil temp critical for S55 due to bearing issues).
  All entries marked `[needs verification]`. ~220 lines.
- `community/profiles/README.md` — add N20/N26/S55 to
  the table of supported engines.
- `CHANGELOG.md` — one bullet per new profile.
- (No `cargo test`, no `node --test` required — TOML
  parse is the only verification, and the
  `community::load()` path already validates all TOML
  files at startup.)

**Pure profile data** — **auto-merge eligible** per
CLAUDE.md rule 2 (no code, only data + docs).

**Acceptance.** Both files parse cleanly via
`python -c "import tomllib; tomllib.load(open(f, 'rb'))"`
for `f in [community/profiles/n20.toml,
community/profiles/s55.toml]`. The
`community::load()` startup log shows them registered.
The profile selector dropdown in the UI surfaces them.

### 🟡 Needs-research items deferred to v0.8.0+

- **Custom math channels** (`map - baro`, `rail / load`)
  — needs a safe expression sandbox; research-grade.
- **Knock detection visualisation (full)** — broader than
  v0.5.0 PR #74's severity-class polish; full distribution
  view + history.
- **AFR / lambda bank readout polish** — decoder exists
  (400B); needs the wider lambda + O2 readiness story.
- **Adaptation / fuel trim readout (full)** — broader than
  v0.5.0 PR #73; per-bank polish.
- **Trigger-based logging** — needs a trigger engine.
- **OBDLink MX+ support** — needs adapter on hand + BLE
  stack work. **Auto-detection in PR #1 unblocks this** —
  once ENET discovery is in, BLE discovery for OBDLink
  follows the same shape.
- **Injector duty cycle** — no separate DID exists per
  v0.6.0 PR #80's discovery; the `inj_time` channel (DID
  `0x4363`, ms) is what the codebase ships. A future
  contributor with F/G-series access can add a
  duty-cycle DID once a real source surfaces.
- **Real-car validation B58 / N55 F-series** — needs
  owner with ENET adapter + F/G chassis. **PR #1's
  auto-detection unblocks this** for any future
  contributor without manual-IP gymnastics. Still
  🟡 until someone with a real car steps up.

### 🟢 Ready items deferred past v0.7.0

- **Export PNG/SVG from charts** — extends the existing
  Chart.js lines. Mid-sized. Good first Chart.js PR.
- **Real-time data logging to disk** — extends the
  existing `LogSession`. Mid-sized. Touches the Logging
  tab only.
- **KWP2000 slow-module timeout fix** —
  `src-tauri/src/protocol/kwp2000.rs`. **Protected path**,
  flag the PR header. Small backend fix; CIC and slow
  modules time out today.
- **ISO-TP multi-frame (FF/CF/FC)** — new module under
  `src-tauri/src/protocol/`. **Protected path**, flag the
  PR header. Required for full VIN and full DTC list over
  UDS.

### Backlog items deferred to v0.8.0+ or later

- Cloud sync (opt-in log upload) — needs privacy + ops
  story first.
- Raspberry Pi CAN bridge — hardware project of its own.
- Plugin system for custom decoders — community governance
  work before code.
- Bootmod3 / MHD integration — legal risk; not appropriate
  scope.
- Multi-language UI — translation coordination problem.
- Web-based shared-log viewer — needs hosted backend work
  first.
- Automatic BMW service manual lookup — needs a corpus.
- Vehicle database (VIN → options, build sheet) — VIN
  decode enrichment.

## Risks and tradeoffs

- **Sibling-agent race** — same multi-writer pattern as
  v0.5.0 / v0.6.0 (already exercised and proven). Always
  work in a fresh sibling clone, branch off `origin/main`,
  push from the clone.

- **`origin/main` drift** — local `main` was stale during
  v0.4 and v0.5 cycles. Always `git fetch origin` before
  basing a branch, and diff against `origin/main`, not
  `main`.

- **CLAUDE.md rule 2** — only doc-only PRs auto-merge.
  PR #1 (ENET/DoIP auto-detection) touches a protected
  path (`src-tauri/src/transport/**`) and **must** go
  through human review. PRs #2 and #3 are doc-only /
  pure data and eligible for auto-merge.

- **GH007 private-email push block** — every commit in
  v0.7.0 should use `ohgeeceee@users.noreply.github.com`
  as the author email (per the v0.6.0 reference
  "Pitfall 6" lesson), not the personal address. The
  cost of using the right email from the start is 5
  seconds; the cost of a push-time 403 + amend +
  force-push is 5 minutes.

- **Discussion-first** — per `COMMUNITY_FRAMEWORK.md`,
  this plan is a seed, not a commitment. Anyone
  (including sibling agents) can argue with it on the
  Discussion thread before any of these PRs land.

- **Real-car evidence still 🟡** — nothing in v0.7.0
  should claim "verified on a real N20 / S55 / F/G"
  unless someone has actually validated it on a real car.
  The N20 / S55 profiles ship with `[needs verification]`
  on every entry, per the v0.5.0 PR #73 discipline.

- **ENET/DoIP discovery has no CI surface** — UDP
  broadcast on port `13400` cannot be tested in a
  sandboxed CI environment. The new tests cover the
  parser (DoIP vehicle identification response → VIN/IP)
  and the activation handshake bytes; the actual UDP
  loop is real-car-only. **Real-car evidence is a
  prerequisite** before this PR merges — see the
  v0.7.0_first_pr doc for the manual verification steps.

## Why "Unblockers" as the cycle name

The v0.4 → v0.5 → v0.6 cycle names were aspirational
("Tuner Friendly" → "Ground Truth" → "Real Hardware"):
each cycle promised a particular user-facing capability
and shipped it. v0.7.0 doesn't promise a new
user-facing capability. It promises to **remove the
friction that contributors and F/G users hit today**:

- F/G users hit the hardcoded-IP wall today (PR #1
  removes it via auto-detection).
- First-time contributors hit the lack-of-easy-PR-items
  wall today (PR #2 consolidates three of them into one
  cycle).
- N20 / S55 owners hit the missing-profile wall today
  (PR #3 ships the missing data).

"Unblockers" is honest about the cycle's actual work:
the value isn't a new feature, it's removing the
obstacles that stand between the next contributor and
their first merged commit, and between the next F/G
user and a working connection.

If a future cycle wants to claim an aspirational name,
v0.8.0 is the natural slot — once ENET auto-detection
unblocks real-car F/G validation and the N20/S55
profiles start collecting `[verified]` PRs from
contributors with those chassis.
