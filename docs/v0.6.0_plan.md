# BeeEmUu v0.6.0 — Cycle Plan

> **Status:** v0.5.0 cycle closed on 2026-07-15 (PR #75 cuts the
> release; pending human review at time of authoring). This
> doc is the proposed v0.6.0 cycle shape for the maintainer's
> review. Per `COMMUNITY_FRAMEWORK.md`'s "no feature without
> a Discussion" rule, **no v0.6.0 work should land before a
> Discussion thread opens on the repo.** This doc is the seed
> for that discussion, not a substitute for it.

## Premise

v0.5.0 finished the **decoder + UI plumbing** (v0.3 / v0.4) and
the **real-car validation harness** (v0.5.0). What's left is
**using those abstractions productively on real data**.

v0.6.0 picks the cycle name **"Real Hardware"** because the work
is about turning the now-validated decoder pipeline into actual
tuner-facing workflows: comparing logs across sessions (before/
after a tune), filling the remaining BMW-specific DIDs, and
extending the protocol surface to the OBD-II layer that the
BMW-specific path sits on top of.

This is the **narrowest defensible scope** consistent with the
3-PR spine shape of v0.4.0 and v0.5.0. Each item is small,
well-bounded, and ships independently. Nothing in this plan is
a blocker for any other item.

## Cycle scope (proposed)

**Three small PRs**, in order. Each is small enough to ship on
its own, each doc-only-or-protected as appropriate.

### PR #1 — Log-merge / comparison

**Files**

- `src/js/main.js` — add a "Compare logs" button next to
  "Export CSV" on the Logging tab. Opens a modal with two
  file pickers + a stats diff per channel (mean shift,
  std-dev shift, max shift).
- `src/js/log_diff.js` (new) — pure helper, takes two
  `LogSeries`-like arrays of `{x, y}` points and returns
  `{meanΔ, stdDevΔ, maxΔ, countΔ}`. Tested via
  `node --test src/js/test/log_diff.test.cjs`.
- `src/css/app.css` — small style block for the diff modal.

**Why first.** The histogram viewer (PR #62) gave the user a
way to see *one* log's distribution; this PR gives the user a
way to see *two* logs' distributions side-by-side, which is
the single most common tuner workflow ("before vs. after my
tune"). Pure client-side, no protocol change.

**Acceptance.** Two logs can be diffed in the UI; per-channel
stats deltas are computed and rendered. Tested via
`node --test` for the math; manually verified in the
simulator for the UI.

### PR #2 — Injector duty cycle (decoder + DIDs)

**Files**

- `community/profiles/{b58,n55}.toml` — add the injector duty
  cycle DID(s). Likely `0x401F` on N55 / `0x4xxx` on B58
  (placeholder until a community source confirms). Marked
  `[needs verification]` per the v0.5.0 discipline.
- `src-tauri/src/data/live.rs` — likely a new decoder
  (`u16_div10` or similar). 8-line change to the existing
  decoder enum + matching arm. Protected path
  (`live.rs` adjacent to `protocol/`) — flag for human review.
- `src-tauri/src/community.rs` — no shape change (TOML
  loader already handles arbitrary DID rows).
- `src-tauri/data/mod.rs` — wire the new decoder into
  `decode_from_str` / `decode_to_str`.
- New `src-tauri/src/data/live.rs` unit tests for the new
  decoder.

**Why second.** The v0.4 / v0.5 "Tuner Friendly" / "Ground
Truth" cycles together deliver the fuel-trim and knock DIDs
but leave out injector duty cycle (the third leg of any
tuner's mental model). The decoder foundation from v0.3 is
ready; only the DID encoding and decoder remain.

**Acceptance.** Profile TOMLs parse, live-data tick returns
non-NaN numbers in the simulator, [needs verification] stays
in the profile label until real-car validation.

### PR #3 — OBD-II PID auto-discovery

**Files**

- `src-tauri/src/protocol/mod.rs` — new `scan_obd2_pids()`
  helper that reads standard service 0x01 PIDs `0x00` to
  `0x7F` and reports which respond. ~30 lines, well-contained.
- `src-tauri/src/commands.rs` — new `#[tauri::command]`
  `list_supported_pids()` that calls the helper and returns a
  `Vec<u8>` of supported PIDs. Protected path (`commands.rs`)
  — flag for human review.
- `src-tauri/src/lib.rs` — register the new command.
- `src/js/main.js` — surface the supported PIDs as a small
  panel on the Vehicle Test tab. Useful diagnostic check
  before a scan ("what does the ECU actually respond to").
- New `src-tauri/src/protocol/mod.rs` unit tests for the
  PID scan loop.

**Why third.** Low-complexity Backlog item, useful diagnostic
utility, gets the Backlog rolling. Closes a single-PR class
the v0.4 / v0.5 cycles explicitly avoided (those cycles
focused on BMW-specific DIDs over UDS; this is the only
standard-OBD-II surface shipped so far).

**Acceptance.** Simulated ECU responds to a known PID
subset (per the simulator's existing OBD-II PIDs); UI shows
the supported list in < 2 seconds on a real car.

## What does NOT ship in v0.6.0 (deferred)

These stay on the v0.6.0+ list per
[`ROADMAP.md`](../../ROADMAP.md), but they're real work and
not part of this cycle's spine:

### 🟡 Needs-research items deferred to v0.7.0+

- **Custom math channels** (`map - baro`, `rail / load`) —
  needs a safe expression sandbox; research-grade.
- **Knock detection visualisation polish** (broader than v0.5.0
  PR #74) — distribution view, history, etc.
- **AFR / lambda bank readout polish** — needs the wider
  lambda + O2 readiness story.
- **Adaptation / fuel trim readout (full)** — broader than
  v0.5.0 PR #73; per-bank polish.
- **Trigger-based logging** — needs a trigger engine.
- **OBDLink MX+ support** — needs adapter on hand + BLE
  stack work.
- **ENET/DoIP auto-detection** — needs protocol state-machine
  work + real-car testing across multiple chassis.
- **Real-car validation B58 / N55 F-series** — needs owner
  with ENET adapter + F/G chassis. Unblocks every other
  🟡 above once it lands.

### Backlog items deferred to v0.7.0+ or later

- Cloud sync (opt-in log upload) — privacy + ops story first.
- Raspberry Pi CAN bridge — hardware project of its own.
- Plugin system for custom decoders — community governance
  work before code.
- Bootmod3 / MHD integration — legal risk; not appropriate
  scope.
- Multi-language UI — translation coordination problem.
- Web-based shared-log viewer — needs hosted backend work.
- Automatic BMW service manual lookup — needs a corpus.
- Vehicle database (VIN → options, build sheet) — VIN
  decode enrichment.

## Risks and tradeoffs

- **Sibling-agent race** — same multi-writer pattern as v0.5.0
  (already exercised and proven). Always work in a fresh
  sibling clone, branch off `origin/main`, push from the
  clone.
- **`origin/main` drift** — local `main` was stale during v0.4
  and v0.5 cycles. Always `git fetch origin` before basing a
  branch, and diff against `origin/main`, not `main`.
- **CLAUDE.md rule 2** — only doc-only PRs auto-merge. PR #1
  (log-diff) is pure JS / CSS and eligible; PRs #2 and #3
  touch protected paths (`live.rs`, `commands.rs`,
  `protocol/mod.rs`) and **must** go through human review.
- **Discussion-first** — per `COMMUNITY_FRAMEWORK.md`, this
  plan is a seed, not a commitment. Anyone (including
  sibling agents) can argue with it on the Discussion
  thread before any of these PRs land.
- **Real-car evidence still 🟡** — nothing in v0.6.0 should
  claim "verified on a real N55 / B58" unless someone has
  actually validated it on a real car.
