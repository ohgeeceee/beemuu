*Draft 2026-07-16, with v0.8.0 "Service Bay" effectively complete (PRs
#1 / #3 / #4 merged as #114 / #115 / #116; PR #2 service functions open
as PR #117 awaiting human merge). This plan lands as a PR directly and

`COMMUNITY_FRAMEWORK.md` Rule 2.*

## Premise

Eight release cycles built the diagnostic spine and the service
workstation: read and clear faults on every module, decode live data,
log and compare it, run service routines with honest verification
status, and cover the mainstream engines. What an owner still cannot do
is the thing that makes ISTA+ a *diagnostic* tool rather than a reader:
**guided fault finding** — take a DTC and walk the tech through a test
plan (check wiring → measure sensor → interpret result → branch →
conclusion). Today BeeEmUu answers "what could this code mean?" from
three flat knowledge bases. It never asks "what did you measure?" — and
that question is the entire job.

The survey below says the extension points are real and the grounding
constraint is the same one this project already lives by: ship only
what in-repo sources support, mark the rest honestly, and make the
contribution path do the growing.

## Survey — the existing guided-fault-finding surface

Three subsystems already ship DTC-keyed knowledge. All three are
**flat**: one lookup in, one static answer out. None can express a
branching test plan.

| Subsystem | Data | Engine | Frontend | Shape today |
|---|---|---|---|---|
| Second Opinion | `community/opinions/*.toml` — **3 files** (2A82, 29E0, P0171), 3 perspectives each | `opinions.rs` — per-DTC query, diy→indie→dealer sort | Tabbed cards in the fault-detail panel (`main.js` `loadOpinion`) | Flat `[[opinion]]` rows: action / cost / time / difficulty / source |
| Community Oracle | `community/oracle/*.json` — **2 files**, 7 fingerprint entries (5 n55, 2 generic) | `oracle.rs` — DTC-set fingerprint, exact + partial match (same engine, overlapping DTC, confidence ≥ 50) | Outcome list per ECU (`query_oracle`) | One lookup → ranked fix categories. No multi-step logic, and the fingerprint model is the wrong shape for one |
| Diagnostic Story | `community/stories/*.toml` — **2 files**, 8 `[[dtc_story]]` entries (3 generic, 5 n55) | `story.rs` — snapshot → narrative | Modal with findings + recommendations | `first_check` is a single string — the seed of a step, not a plan |

Adjacent assets the walkthrough can lean on:

- **Fault-detail panel composition** (`main.js:547-553`): clicking a
  DTC already loads freeze frame + opinion + schematics in parallel —
  the walkthrough mounts here, no new navigation concept needed.
- **Freeze frames per DTC**: `protocol` readFreezeFrame + schema decode
  (`data/freeze.rs`, `community/freeze_schemas.toml`) — rpm / coolant /
  speed / load / volts / mileage at time of fault. A test plan can seed
  its first branch from these values ("coolant was 20°C at fault time —
  a cold-engine code").
- **Live-data measurement**: profile DIDs + `readDataByIdentifier` are
  the "measure" verb of every real test plan. Grounded examples in-repo:
  N55/S55 fuel trims `0x1201`/`0x1202` (`TECH_SPECS.md` § 5), the OBDb
  verified DIDs of `research/bmw_diag_dim04_uds_dids.md` (rail pressure,
  wheel speeds, gear, vehicle speed), KWP local IDs
  (`research/bmw_diag_dim07_local_ids.md`).
- **DTC corpus**: 208 overlay entries post-#114
  (`community/dtc_texts.toml`), plus ~450 vetted seed rows
  (`backend/seed_dtcs.py`, `backend/seed_bmw_dim01.py`) and the
  source-URL'd descriptions in `research/bmw_diag_dim01_dtcs.md` — the
  frequency/evidence base for choosing which DTCs get plans first.
- **Parse gate**: every shipped community **TOML** is CI-gated
  (`community.rs::shipped_community_tomls_parse`, v0.8.0 PR #1).

### Survey findings — extension points and constraints

1. **New data category, not a schema break.** Branching plans belong in
   a new `community/testplans/<dtc>.toml` tree with `[[step]]` tables
   (step id, instruction, measurement — DID-poll or manual check —
   expected result, `on_pass` / `on_fail` branch targets, conclusion
   nodes). Extending the flat `[[opinion]]` schema would corrupt the
   perspective-card UI; a parallel tree keeps all three existing
   surfaces byte-stable.
2. **TOML inherits the gate for free.** The v0.8.0 PR #1 walker parses
   every shipped community TOML; a `testplans/` subtree is covered the
   day it lands. A schema-shape test (like
   `shipped_dtc_texts_parse_and_nonempty`) asserts branch integrity:
   every `on_pass`/`on_fail` target resolves to a step in the same file,
   and every plan has at least one conclusion.
3. **The Oracle JSON is ungated — fix in passing.** A broken
   `community/oracle/*.json` fails only as a startup `eprintln`; CI
   sees nothing. The cycle that adds a third JSON-adjacent subsystem
   should extend the gate to JSON first (test-only change).
4. **The Oracle engine stays a pattern matcher.** Multi-step logic
   lives in the plan interpreter, not in `oracle.rs` — its fingerprint
   model answers "what fixed cars like this", a different question from
   "what do I check next".
5. **Conservative sourcing caps plan authorship.** Plans ship only for
   DTCs grounded in-repo: the three subsystems' own source-URL'd
   content, `research/bmw_diag_dim01_dtcs.md` descriptions, dim04/dim07
   measurement DIDs, `TECH_SPECS.md`, and the backend seed corpora.
   That grounds roughly the first **8–10 DTCs** (2A82, 29E0, 29E1,
   29E2, 30FF, 29CC, 2E81/2E82, P0171, P0300, P0420) — the
   highest-frequency set across the corpus. Everything else is a
   documented known-missing list, same discipline as v0.8.0 PR #2.

## Milestone theme

**"Guided Fault Finding"** — turn the flat knowledge bases into a
walkthrough: open a fault, get a test plan, measure what it asks,
branch on the result, land on a conclusion that cites the same
community evidence the project already ships. The honesty rules don't
move: no invented procedures, every step cites its in-repo source, and
the plan corpus grows through contributions with evidence.

## PR-sized slices

Five PRs, mirroring prior cycles' shape. Tier per `CLAUDE.md`.

### PR #1 — Test-plan schema + parse-gate extension (Tier A)

**Why first.** The schema is the contract every later slice builds on,
and the gate must exist before plan files pile up — the exact lesson of
v0.8.0 PR #1.

**Files**

- `docs/testplans.md` (new) — the `[[step]]` schema spec: step ids,
  instruction text, measurement kinds (`did:<hex>` live-data poll with
  expected range; `manual` observation with pass/fail question),
  `on_pass` / `on_fail` targets, conclusion nodes (with a `conclusion`
  text + optional link into the matching opinion/oracle entry), and the
  mandatory per-step `source` citation rule.
- `src-tauri/src/community.rs` — **test-only**: extend the shipped-TOML
  gate to walk `community/testplans/` (it already walks the whole tree
  — the shape test is the real work) and add
  `shipped_testplans_branch_integrity`: every branch target resolves,
  every plan ends in ≥ 1 conclusion, every step names a source. Add a
  JSON parse gate for `community/oracle/*.json` (survey finding #3).
- `community/testplans/README.md` — author-facing mirror of the schema
  doc, in the style of `community/profiles/README.md`.
- `CHANGELOG.md`.

**Acceptance.** A deliberately broken fixture plan (dangling branch,
missing conclusion, unsourced step) fails `cargo test` — proven in the
PR with a temporary fixture, then removed. `lint-toml.js` clean. No
production-code change.

## What we will NOT do (and why)

Honesty is a feature. Guided diagnostics sits next to some legally and
ethically fraught territory; these stay out:

- **Emissions-monitor tampering.** No guided content for clearing
  readiness monitors, defeating catalyst/O2 diagnostics, or "pass
  inspection" workflows. A P0420 plan ends at *diagnose the leak or the
  cat* — never at how to mask it. **Permanent exclusion.**
- **VIN spoofing / odometer correction.** Fraud-adjacent, no legitimate
  owner use case. **Permanent exclusion.**
- **Imported ISTA test plans.** ISTA's guided test-plan corpus is
  copyrighted BMW material — same exclusion as the repair-instruction
  corpus in v0.8.0. Community-sourced, evidence-cited plans are the
  ceiling.
- **Auto-executing plan steps.** A plan never fires a write
  (routineControl, clear-DTC) on its own; write steps hand off to the
  existing service-function UI with its `[UNVERIFIED]` gating intact.
  Guidance reads; humans write.
- **LLM-generated procedures shipped unreviewed.** `story.rs` already
  notes a possible v2 LLM layer. Any future generated content enters
  the corpus only through the same source-citation + harness-validation
  path as hand-authored plans — generation is drafting, not grounding.
- **Cloud upload of fault sets.** The Oracle is deliberately offline
  and privacy-preserving (no VIN in fingerprints); guided diagnostics
  inherits that. The anonymised snapshot-share path remains the only
  sharing mechanism.

## Risks and tradeoffs

- **Procedural temptation.** PR #2 is where a well-meaning contributor
  pastes a forum repair sequence as a "test plan". The per-step
  `source` citation rule + the known-missing list are the whole design;
  the PR template should name the source per plan.
- **Schema over-fit.** A branching graph can grow cycles, diamonds, and
  spaghetti. The PR #1 integrity gate enforces resolvable branches but
  deliberately does *not* forbid DAG shapes; linear-with-conclusions is
  the authoring guidance, not a hard rule, until real plans show what
  breaks.
- **Scope creep into the Oracle.** Extending `oracle.rs` with multi-step
  logic would entangle two different questions (pattern match vs guided
  procedure). The plan interpreter stays separate; the walkthrough only
  *links* oracle outcomes at conclusion time.
- **Tier B surface is small but real.** PR #3 touches `commands.rs`;
  keeping the command read-only and the traversal frontend-side is what
  keeps the review surface tiny. The `async_commands.rs` allowlist edit
  is a contract change and is called out as one.
- **`origin/main` drift** — same discipline as prior cycles: fetch and
  branch off `origin/main`, never local `main`.
- **GH007 private-email push block** — author commits as
  `ohgeeceee@users.noreply.github.com` from the start.
- **Direct-PR model** — per `COMMUNITY_FRAMEWORK.md` Rule 2, this plan
  lands as a PR on its own; slices dispatch as PRs when the work is
  done. No Discussion thread gates any of them.

## Why "Guided Fault Finding" as the cycle name

Cycle names track the user promise. v0.8.0 made BeeEmUu the service
bay; v0.9.0 makes it the diagnostician. The name is deliberately the
ISTA+ feature it closes the gap on — and honest about the ceiling:
community-sourced, evidence-cited, harness-validated plans, not a
300-MB proprietary procedure database.
